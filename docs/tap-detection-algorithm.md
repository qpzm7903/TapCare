# 敲打检测算法文档

## 1. 硬件约束

### 传感器

| 项目 | 值 | 备注 |
|------|-----|------|
| 传感器 | `@system.sensor` subscribeAccelerometer | GT6 内置加速度计 |
| 可用 interval | `'normal'` 唯一可用 | `'game'`、`'ui'` 静默失败（0 回调） |
| 实际采样率 | ~5Hz（每帧 ~200ms） | 硬件/固件限制，无法提升 |
| 数据单位 | g（去重力后线性加速度） | 不是 m/s² |
| 静止 magnitude | ≈0.25g | 噪声波动 0.1~0.3g |

### 陀螺仪（备用，未集成）

- `subscribeGyroscope` 可用，需 `ohos.permission.GYROSCOPE` 权限
- 同样 ~5Hz 采样率
- 详见 `gyroscope-fusion-plan.md`

### 5Hz 采样的物理限制

一次敲打的加速度脉冲宽度约 50~150ms，采样间隔 200ms。根据脉冲宽度，采样点命中概率：

| 力度 | 脉冲宽度 | 命中概率 |
|------|----------|----------|
| 轻敲 | ~50ms | ~25% |
| 中敲 | ~100ms | ~50% |
| 重敲 | ~150ms+ | ~75% |

**结论：5Hz 下轻敲必然有一定漏检率，这是物理极限。**

## 2. 算法总览

整个检测流程分三阶段：**Baseline 校准 → 信号提取 → 上升沿判定**。

```
传感器回调 (x, y, z) 每 ~200ms
        │
        ▼
┌─ Baseline 校准（前 10 帧，约 2 秒）─┐
│  采集样本 → 排序 → 取最小 6 个均值   │
│  → 得到静态 baseline ≈ 0.20g        │
└──────────────────────────────────────┘
        │ 校准完成
        ▼
┌─ 信号提取 ──────────────────────────┐
│  dynamicAccel = |magnitude - baseline|│ ← 绝对偏移
│  delta = |magnitude - prevMagnitude|  │ ← 帧间差值
└──────────────────────────────────────┘
        │
        ▼
┌─ 上升沿判定 ────────────────────────┐
│  dynHit = dynamicAccel >= 0.18g      │
│  deltaHit = delta >= 0.15g           │
│  if (dynHit OR deltaHit)             │
│    且 _tapFired == false             │
│    且 since >= 150ms                 │
│    → 触发计数，_tapFired = true      │
│  else if 信号回落                    │
│    → _tapFired = false（准备下一次） │
└──────────────────────────────────────┘
```

---

## 3. 各阶段详解

### 3.1 Baseline 校准

**目的**：测量手表静止时的加速度 magnitude 基准值。

**流程**：
1. 开始计数后，前 10 帧（约 2 秒）只采集不检测
2. 手写冒泡排序（Lite Wearable 不支持 `Array.sort`）
3. 取最小的 6 个样本计算均值，作为 baseline
4. 用最小 N 个而非全部均值，是为了排除校准期间用户已开始敲打的干扰样本

**参数**：
- `CAL.SAMPLE_COUNT = 10`（采集帧数）
- `CAL.USE_LOWEST = 6`（取最小几个）

**校准完成后的初始化**：
- `_prevMagnitude = magnitude`（用最后一帧初始化，防止首帧 delta 异常）
- `_lastTapTime = now`（防止首帧 since 为天文数字导致误触发）

### 3.2 信号提取：双通道检测

每帧计算两路信号：

#### 通道 A：绝对偏移 (dynamicAccel)
```
dynamicAccel = |sqrt(x² + y² + z²) - baseline|
```
- 捕获**采样点恰好落在脉冲峰值上**的情况
- 阈值 `TAP_THRESHOLD = 0.18g`

#### 通道 B：帧间差值 (delta)
```
delta = |当前 magnitude - 上一帧 magnitude|
```
- 捕获**采样点落在脉冲边缘**的情况
- 即使没采到峰值，敲打前后的帧 magnitude 也会有跳变
- 阈值 `DELTA_THRESHOLD = 0.15g`

#### 触发逻辑
```
触发 = dynHit OR deltaHit
信号强度 signal = max(dynamicAccel, delta)
```

日志中标注触发原因：
- `DYN`：仅绝对偏移触发
- `DELTA`：仅帧间差值触发（捕获了绝对偏移漏掉的敲打）
- `BOTH`：双重触发（最可靠）

### 3.3 上升沿判定

用布尔标志 `_tapFired` 实现"一次敲打只计一次"：

| 状态 | 条件 | 行为 |
|------|------|------|
| 待触发 | `_tapFired == false` 且信号越阈值 且 `since >= 150ms` | 触发计数，设 `_tapFired = true` |
| 已触发 | `_tapFired == true` 且信号越阈值 | 仅跟踪峰值，不重复计数 |
| 复位 | 信号回落到两个阈值以下 | 设 `_tapFired = false` |

额外 150ms debounce 防止极端抖动下的连续误触。

### 3.4 力度分级

基于触发时的 signal 值：

| 力度 | 范围 | 真机参考 |
|------|------|----------|
| 轻 | < 0.4g | 轻敲手腕 |
| 中 | 0.4g ~ 1.0g | 正常敲打 |
| 重 | > 1.0g | 用力敲击 |

### 3.5 节奏评估

记录最近 10 次敲打间隔，计算变异系数 (CV = 标准差/均值)：

| CV | 评价 |
|----|------|
| < 0.15 | 很稳 |
| 0.15 ~ 0.30 | 良好 |
| > 0.30 | 不稳 |

## 5. 算法演进历史

| 版本 | 方案 | 问题 |
|------|------|------|
| v1 | 状态机（上升-峰值-下降） | 5Hz 下一次敲打只有 1~2 帧，状态机无法追踪完整波形 |
| v2 | 简单阈值 + 时间 debounce | 阈值 0.5g 太高，轻敲漏检严重 |
| v3 | 降低阈值 + baseline 校准 | 改善但仍依赖单一绝对值 |
| v4 | 上升沿检测替代时间 debounce | 防止持续振动被计多次 |
| **v5（当前）** | **双通道：绝对偏移 + 帧间差值** | 覆盖"采到峰值"和"采到边缘"两种情况 |

## 6. 后续优化方向

### 方向 A：陀螺仪融合（已验证可行，待实施）

**原理**：同时订阅加速度计和陀螺仪，要求两路信号同时异常才算敲打。

**收益**：
- 可以更安全地降低阈值（双重确认减少误检）
- 走路、手势等干扰通常只触发一路

**挑战**：
- 两路回调时间可能不完全同步，需做时间窗口对齐
- 额外传感器可能增加功耗
- 陀螺仪的 game 模式未测试

**详见**：`gyroscope-fusion-plan.md`

### 方向 B：节奏预测补偿

**原理**：检测到稳定敲打节奏后（如每 400ms 一次），如果某个预期位置漏检，可推断补一次。

**适用场景**：经络敲打本身节奏规律，假设合理。

**实现思路**：
```
已有节奏均值 avgInterval
预期下一次敲打时间 = lastTapTime + avgInterval
如果超过 expectedTime + 容差 仍未检测到 → 补计一次
```

**风险**：
- 用户暂停或变速时可能误补
- 需要至少 3~5 次连续稳定敲打才能建立节奏预测
- 补计的力度无法评估（没有真实信号）

### 方向 C：自适应阈值

**原理**：根据近期信号的噪声水平动态调整阈值，而非使用固定值。

**实现思路**：
```
维护滑动窗口（最近 20 帧）的 dynamicAccel 均值和标准差
自适应阈值 = 均值 + K × 标准差（K=2~3）
```

**收益**：
- 适应不同佩戴松紧度、不同运动状态
- 手表静止时阈值自动降低（更灵敏），运动时自动升高（防误触）

**风险**：
- 持续敲打会拉高均值和标准差，导致阈值越来越高
- 需要区分"敲打帧"和"静止帧"来维护窗口

### 方向 D：信号积分 / 能量窗口

**原理**：不看单帧峰值，而是看连续 2~3 帧的能量总和。

**实现思路**：
```
energy = sum(dynamicAccel²) over sliding window of 3 frames
if energy >= ENERGY_THRESHOLD → 触发
```

**收益**：
- 即使单帧没采到明显峰值，周围帧的残余能量累加可能超过阈值
- 对 5Hz 的低采样率更鲁棒

**风险**：
- 3 帧窗口 = 600ms，可能导致相邻两次快速敲打合并为一次
- 触发时间点不精确，力度评估不准

### 方向 E：硬件层面

**可能性极低，但值得记录**：
- 未来固件更新可能开放更高采样率
- 其他传感器（气压计、心率传感器的原始信号）可能提供辅助判据
- 如果有陀螺仪的 game 模式可用（未测试），可以用高频陀螺仪辅助

### 优先级建议

| 优先级 | 方向 | 理由 |
|--------|------|------|
| 1 | C 自适应阈值 | 改动小，不依赖额外传感器，直接提升适应性 |
| 2 | A 陀螺仪融合 | 已验证可行，能显著减少误检 |
| 3 | B 节奏预测 | 适合经络敲打场景，但逻辑复杂 |
| 4 | D 能量窗口 | 理论有价值，但与快速敲打有冲突 |
