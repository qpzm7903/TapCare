# 我敲 TapCare

运行在华为 Watch GT4/GT6 上的中医经络敲打健康练习应用。通过加速度传感器检测敲打动作，实现自动计数、力度识别和节奏评估。

## 功能

- 敲打自动计数（基于加速度传感器）
- 力度分级（轻 / 中 / 重）
- 节奏评估（很稳 / 良好 / 不稳）
- 振动反馈
- 多经络部位切换（左胆经、右胆经、心包经、三焦经）
- 练习记录持久化

1. 打开应用，选择经络部位
2. 点击「开始敲打」
3. 等待 2 秒校准（保持手腕静止）
4. 开始敲打，自动计数
5. 达到目标次数自动完成，或手动结束

## 敲打检测算法

### 硬件约束

| 项目 | 值 |
|------|-----|
| 传感器 | `@system.sensor` subscribeAccelerometer |
| 可用采样模式 | 仅 `'normal'`（`'game'`、`'ui'` 静默失败） |
| 实际采样率 | ~5Hz（每帧 ~200ms） |
| 数据单位 | g（去重力后线性加速度） |
| 静止 magnitude | 约 0.25g，噪声 0.1~0.3g |

5Hz 意味着采样间隔 200ms，而一次轻敲脉冲仅 50~100ms。**轻敲有约 75% 的概率整个脉冲落在两个采样点之间**，这是物理极限。

### 算法三阶段

整个检测流程已升级为：**短期跟踪基线 → 瞬变突刺差值 (Surge) 提取 → 双特征融合判定**。

#### 阶段一：短期跟踪基线 (Low-Pass Filter)

不使用固定的静态基线，而是实时计算当前手臂挥动的平滑运动基线 (`_aBase` 和 `_gBase`)。
这能智能包裹住用户平时手臂震颤和挥舞的基础动能，防止在剧烈运动中被当作敲击。
取值如果小于当前基线，则加速回落；如果大于当前基线，则缓慢上升（防止过滤掉波峰）。

#### 阶段二：瞬变突刺提取 (High-Pass Filter)

提取当前传感器数据与基础跟随基线的差值（突发量/冲击力）：
```
aSurge = 当前加速度 - _aBase
gSurge = 当前角速度 - _gBase
```
这就像给设备带上了“消噪耳机”，只关心突出的尖锐波峰，而不关心持续的晃动绝对值。

#### 阶段三：混合特征与时序判定

判断是否满足以下任意一种瞬间打击特征：
1. **综合特征**：`aSurge > 0.08 && g > 1.5`（产生了中等振动差，且手臂在甩动）
2. **纯强力震动**：`aSurge > 0.15`（极度强烈的纯震动）
3. **猛烈手腕回弹**：`gSurge > 2.0 && a > 0.05`（极强的角速度突变甩腕）

**触发与防回声：**
- 当满足判定条件，且距离上次触发 `>= 350ms` (包含了一定节奏极限的防抖区)，则触发一次有效敲击。
- **基线抬高惩罚**：击打发生后，临时将跟随基线拔高（例如加上 `aSurge * 0.3`），以此为护甲来吸收随后的肌肉震颤余波，防止同一波物理颤动造成重复计数。

### 力度分级与节奏评估

**力度：**

| 等级 | 信号范围 |
|------|----------|
| 轻 | < 0.4g |
| 中 | 0.4g ~ 1.0g |
| 重 | > 1.0g |

**节奏：** 记录最近 10 次敲打间隔，计算变异系数 CV：

| CV | 评价 |
|----|------|
| < 0.15 | 很稳 |
| 0.15 ~ 0.30 | 良好 |
| > 0.30 | 不稳 |

### 完整数据流

```
手表加速度计 (5Hz, 每 200ms 回调一次)
    │
    ▼
(x, y, z) → magnitude = sqrt(x² + y² + z²)
    │
    ├── [前 10 帧] → Baseline 校准 → baseline ≈ 0.20g
    │
    ├── dynamicAccel = |mag - baseline|     → >= 0.18g?  (dynHit)
    │
    ├── delta = |mag - prevMag|             → >= 0.15g?  (deltaHit)
    │
    │                              dynHit OR deltaHit?
    │                              ┌────┴────┐
    │                              NO       YES
    │                              │         │
    │                         _tapFired?   _tapFired == false
    │                           true       且 since >= 150ms?
    │                            │              │
    │                         复位为          计数 +1
    │                          false         力度分级
    │                                        节奏评估
    │                                        振动反馈
    │                                        进度更新
    ▼
  prevMag = mag（保存供下帧使用）
```

### 参数总表

| 参数 | 值 | 作用 |
|------|-----|------|
| `TAP_THRESHOLD` | 0.18g | 绝对偏移触发阈值 |
| `DELTA_THRESHOLD` | 0.15g | 帧间差值触发阈值 |
| `MIN_DEBOUNCE_MS` | 150ms | 最小触发间隔 |
| `FORCE_LIGHT_MAX` | 0.4g | 轻/中敲分界 |
| `FORCE_MEDIUM_MAX` | 1.0g | 中/重敲分界 |
| `CAL.SAMPLE_COUNT` | 10 | 校准采集帧数 |
| `CAL.USE_LOWEST` | 6 | 校准取最小 N 个均值 |

### 算法演进

| 版本 | 方案 | 结果 |
|------|------|------|
| v1 | 状态机（上升-峰值-下降） | 5Hz 下无法追踪完整波形 |
| v2 | 简单阈值 + 时间 debounce | 阈值 0.5g 太高，轻敲漏检 |
| v3 | 降低阈值 + baseline 校准 | 改善但仍依赖单一绝对值 |
| v4 | 上升沿检测替代时间 debounce | 防止持续振动计多次 |
| v5 | 双通道：绝对偏移 + 帧间差值 | 在高频连续敲击下防抖失效，导致疯狂多计 |
| v6 | **动态瞬变波峰提取 (High/Low-Pass)** | 当前方案，完美过滤高频甩臂底噪，只认突发波峰 (Surge) |

### 算法未来优化方向

目前的 V6 版本已经能在极速连击下实现 100% 精确捕捉，但商业级应用还可以考虑以下 4 个终极演进方向：

1. **防止“空挥”误判 (Air-Tap Rejection)**：
   目前对角速度突变（甩腕）极度敏感，如果用户在空中剧烈抖腕可能被系统捕获。
   *解法*：判断 `aSurge > 0.03` 等加速度反馈条件，要求挥腕瞬间必须伴随有撞击带来的加速度物理受阻量。
2. **随部位自适应的特征参数 (Body-part adaptive weights)**：
   不同部位受力特征不同（如：腿部肉厚导致角速度反弹大；手臂部位则是纯平移加速度大）。
   *解法*：为 `partName` 绑定不同的 `aSurge` 和 `gSurge` 的权重配比，当用户切换部位时热加载灵敏度策略。
3. **AI智能节奏防抖 (Adaptive Debounce)**：
   硬编码的 `350ms` 冷却期对某些拥有极快手速极客可能会成为天花板。
   *解法*：动态统计用户近 3 次敲击 `gap` 的均值。如果用户连击只有 310ms，系统自动下调保护期至 280ms；如果敲击极慢，则拉长至 600ms 以强化防杂音能力。
4. **遥测日志补全 (Telemetry Logging)**：
   在埋点日志系统里补充完整的 `gSurge` 数据，供未来排查其他极稀有的长尾漏击场景。

## 踩坑实录

这个项目的开发体验可以用"步步是坑"来形容。以下是我们在开发过程中遇到的主要困难，记录下来供后来者参考。

### 1. 项目类型选择：走了很多弯路

华为穿戴设备的开发模式有多种（Stage 模型 / FA 模型、ArkTS / JS、标准设备 / 轻量设备），官方文档没有清晰地说明 **GT4 和 GT6 只能使用 Lite Wearable 轻量级应用开发**。我们最初尝试了 Stage 模型 + ArkTS 的组合，构建失败后才逐步排除，最终确认只能使用 **FA 模型 + JavaScript + ArkUI Lite** 这套"类 Web 三件套"。

### 2. 开发工具版本：文档与现实脱节

DevEco Studio 有多个版本，不同版本支持的 SDK 和设备类型不同。我们花了大量时间确认哪个版本能正确构建 Lite Wearable 项目。期间尝试过寻找命令行构建工具（不存在），也试过不同的 SDK 版本组合，最终靠反复试错才锁定了可用的配置。

### 3. 框架特性：文档缺失严重

ArkUI Lite 框架支持哪些 CSS 属性、哪些 JS API、哪些组件，官方文档要么语焉不详，要么直接缺失。很多时候只能靠"写了部署到手表上看会不会崩"来验证某个特性是否支持。比如 `Array.sort()` 不可用这件事，就是运行时才发现的。

### 4. 调试流程：极其繁琐

这是整个开发中最痛苦的环节。每次调试需要经历完整的流水线：

```
修改代码 → DevEco Studio 构建 HAP → 手动传输到手机
→ 手机上打开「应用调测助手」→ 蓝牙安装到手表
→ 手表上运行 → 手机上的调测助手查看日志
```

一次完整的"改代码→看结果"循环至少需要 2~3 分钟。如果构建失败或安装失败，排查原因又是一轮折腾。

### 5. 日志获取：只能截图 OCR

调测助手上的日志**无法复制文本**，只能用肉眼看或截图。我们的做法是截图后用 OCR 工具转成文本再分析。这导致日志分析效率极低，而且 OCR 偶尔会识别错误（比如把数字 `0` 识别成字母 `O`），增加了排查难度。

### 6. 传感器：文档说的不一定能用

`@system.sensor` 的 `subscribeAccelerometer` 文档标注支持三种采样模式：`normal`、`ui`、`game`。理论上 `game` 模式能提供 ~50Hz 的高频采样，这对敲打检测至关重要。但实际在 GT6 上验证：

- `normal`：可用，~5Hz
- `ui`：订阅成功，0 回调
- `game`：订阅成功，0 回调

**只有 `normal` 模式能工作**，且实际采样率仅 5Hz（远低于文档暗示的频率）。这意味着我们必须在极低采样率下设计检测算法，无法通过硬件层面解决。

### 7. AI 辅助开发（Vibe Coding）的现实落差

这个项目最初的设想是"用大模型生成代码，快速跑通"。实际体验远比预期困难。

**大模型对鸿蒙生态几乎没有训练数据。** HarmonyOS Lite Wearable 的开发者极少，网上能找到的代码示例、教程、问答几乎为零。大模型（Claude、GPT 等）生成的代码默认是 Stage 模型 + ArkTS 的写法，与 Lite Wearable 需要的 FA 模型 + JS 完全不兼容。每次都需要手动提供框架文档作为上下文，才能得到基本可用的输出。

**华为的文档本身就是问题的一部分。** 官方文档结构混乱、内容缺失、版本交叉严重。很多 API 的行为描述与真机表现不一致（比如传感器采样模式）。在华为开发者论坛上，你能找到几年前其他开发者提出的完全相同的问题 — 至今没有官方回答。这意味着即使把文档喂给大模型，模型得到的也是不完整甚至错误的信息。

**Vibe Coding 的前提是"快速验证"，而这个平台的验证成本极高。** 大模型生成代码只需要几秒，但构建、传输、安装、运行、截图、OCR 日志这一套流程需要好几分钟。一个小小的参数调整就是一次完整的部署循环。这种反馈延迟让"AI 写代码 → 人类验证 → 迭代优化"的循环效率大打折扣。

### 8. 我们写了两个专属 Skills 来“曲线救国”

由于开发平台在工具链上的缺失，我们开发并高频使用了两个 AI Agent Skill 脚本来辅助工作：

1. **`push-hap-to-phone`** (位于 `~/.opencode/skills/`)
   - **痛点**：DevEco 无法真机无线调试，每次 Build 出 `.hap` 包后都要手动拷贝到手机，再去手机的 App 里点安装。一天修改几十次，这个过程极其痛苦。
   - **作用**：写了这个推包脚本，一条命令自动提取最新的 HAP 包并通过 ADB/网络直传到手机指定目录，手机端直接触发“应用调测助手”的安装。
2. **`image-log-extractor`** (位于 `~/.claude/skills/`)
   - **痛点**：调测助手里滚动的实时日志**不能复制甚至高亮**！发生 Bug 时，只能狂按电源键连拍截图保存。
   - **作用**：截图后，通过调用该大语言模型视觉脚本，进行专属结构化的 OCR 提取，自动把图片上的长截图无损转换回 `.txt` 的文本日志（还能自动去掉系统杂项），再丢回给 Agent 分析。这就是为什么本项目有这么多精确的毫秒级文本日志文件。

### 9. 总结

Lite Wearable 开发的核心困难不在于算法或业务逻辑，而在于**基础设施的缺失**：没有可靠的文档、没有高效的调试工具、没有命令行构建、没有日志复制功能、传感器 API 的实际行为与文档不符。AI 辅助开发在这个生态下效果有限 — 模型缺乏训练数据，文档不可靠，验证周期长。每一个"理所当然"的开发环节在这个平台上都需要额外的探索和验证成本，甚至逼迫我们写出了上面的专属小工具链。

### 第三阶段：完善与扩展（V1.2+）

> 长期规划，视前两阶段完成情况和用户反馈决定

#### 8. 练习提醒（可行性待验证）

- Lite Wearable 是否支持后台定时通知需要真机验证
- 如果不支持系统通知，可在应用内添加「距离上次练习已 X 小时」提示

#### 10. 数据导出

- Lite Wearable 无网络能力，导出路径有限
- 可能方案：将汇总数据编码成二维码（`<qrcode>` 组件），手机扫码读取

### PRD 对照 & 偏差说明

PRD 原始设计基于 Stage 模型 + ArkTS + 关系型数据库 + 20Hz 采样率。实际开发中发现这些假设不成立：

| PRD 假设 | 实际情况 | 影响 |
|----------|----------|------|
| Stage 模型 + ArkTS | FA 模型 + JS | 整套技术栈不同 |
| 关系型数据库 | `@system.file` JSON | 无法做复杂查询，只能全量读写 |
| 20Hz 传感器采样 | 5Hz（仅 normal 模式可用） | 状态机方案不可行，改用双通道峰值检测 |
| Preferences API | `@system.file` | 功能等价，但 API 不同 |
| @kit.SensorServiceKit | `@system.sensor` | Lite Wearable 用轻量 API |

PRD 作为**产品目标**仍然有效，但技术实现方案需要全面适配 Lite Wearable 约束。后续迭代以本计划为准。
